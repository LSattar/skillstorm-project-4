MVP SPEC — Animal Shelter Management Platform

Tech Stack
- Backend: Java 17, Spring Boot (Web, Validation, Spring Data JPA, Spring Security)
- Auth: JWT + optional Google OAuth2 login
- Database: MySQL 8
- Frontend: Angular
- Object Storage: S3

Goal
Build a full-stack platform to manage animals through intake → shelter placement → foster placement → adoption, with:
- Public browsing of animals
- Authenticated adopters can apply to adopt
- Authenticated staff can intake/move animals and review/approve/deny applications
- Audit/history of important animal events

Roles & Permissions (MVP)
- PUBLIC (not logged in)
  - Browse/search animals, view animal details
- ADOPTER (authenticated)
  - Create/update own profile + questionnaire
  - Submit adoption applications for animals
  - View status of own applications
- STAFF (authenticated; includes shelter employees)
  - CRUD animals (intake, update, deactivate)
  - Move animals (between shelter locations or to foster)
  - Review adoption applications; approve/deny
  - Complete adoption (finalize) and update animal status
- FOSTER (optional as separate role for MVP; can be merged into ADOPTER or STAFF if needed)
  - View foster assignments; update limited notes (optional)

MVP Scope Boundaries
IN SCOPE
- Auth (local username/password) + password hashing
- JWT-based session for frontend API calls
- Optional: Google OAuth2 login (link to a user record)
- Animal CRUD, basic photo support, search/filter
- Adoption application submission + staff review workflow
- Event/history tracking for animal lifecycle
- Basic shelter location support (at least 1 location; multi-location supported if time)

OUT OF SCOPE (for MVP)
- Payment processing
- Messaging/chat, email/SMS notifications
- Scheduling/appointments
- Complex medical record systems (beyond simple flags/notes)

Core Data Model (MySQL Tables / Entities)

- has_kids (boolean)

- needs_good_with_other_pets (boolean)
- willing_medically_complex (boolean)

4b) adopter_questionnaires
(stores latest questionnaire answers for each adopter)

- id (PK)
- user_id (FK users.id, unique)
- schema_version (int)
- household_size (int)
- housing_type (enum/string: HOUSE/APARTMENT/CONDO/OTHER)
- has_yard (boolean)
- has_kids (boolean)
- has_other_pets (boolean)
- needs_good_with_kids (boolean)
- needs_good_with_other_pets (boolean)
- willing_medically_complex (boolean)

- created_at
- updated_at

5) shelters (or locations)
- id (PK)
- name
- address fields (optional)
- capacity_total (optional)

6) animals

- name

7) animal_photos (stored in S3 bucket ljs-animal-shelter-photos)
- id (PK)

8) animal_events
(tracks history of lifecycle events)
- id (PK)
- animal_id (FK animals.id)

- to_shelter_id (FK shelters.id, nullable)
- from_foster_user_id (FK users.id, nullable)
- to_foster_user_id (FK users.id, nullable)
- performed_by_user_id (FK users.id, nullable)  // staff actor, or adopter for application submit
- notes (text, nullable)
- occurred_at (datetime)

9) adoption_applications
- id (PK)
- animal_id (FK animals.id)
- adopter_user_id (FK users.id)
- status (enum/string): SUBMITTED, IN_REVIEW, APPROVED, DENIED, WITHDRAWN
- questionnaire_snapshot_json (text/json)  // store answers at time of submission
- staff_reviewer_user_id (FK users.id, nullable)
- decision_notes (text, nullable)
- created_at, updated_at

10) adoptions
(finalized adoption record)
- id (PK)
- animal_id (FK animals.id)
- adopter_user_id (FK users.id)
- application_id (FK adoption_applications.id)
- adopted_at (datetime)
- finalized_by_user_id (FK users.id)
- notes (text, nullable)

Goal:
- Keep a minimal, consistent set of fields that match the core adopter_profile data.
- Questionnaire answers are stored in adopter_questionnaires.
- A subset of fields are synchronized into adopter_profiles for filtering and review.
- Applications store a frozen snapshot of questionnaire answers at submission time (questionnaire_snapshot_json).
- Questionnaire submission updates adopter_profile automatically.
- Applications store a frozen copy of the questionnaire answers at submission time (questionnaire_snapshot_json).

Questionnaire Fields (v1)

Contact / Household
- householdSize (int)
- phone (string)
- city (string)
- state (string)
- zip (string)

Home Environment
- housingType (enum: HOUSE, APARTMENT, CONDO, OTHER)
- hasYard (boolean)
- hasKids (boolean)
- hasOtherPets (boolean)

Pet Fit / Constraints (aligns with animal filters like goodWithKids/goodWithOtherPets/medicallyComplex)
- needsGoodWithKids (boolean)
- needsGoodWithOtherPets (boolean)
- willingMedicallyComplex (boolean)

Notes
- notes (string, optional)

Mapping Rules (Questionnaire -> Profile/User)

On PUT /api/adopter/questionnaire:

- Upsert adopter_questionnaires for the current user.
- Update users.phone if phone is provided.
- Sync overlapping fields into adopter_profiles:

  household_size            = householdSize
  housing_type              = housingType
  has_yard                  = hasYard
  has_kids                  = hasKids
  has_other_pets            = hasOtherPets
  needs_good_with_kids      = needsGoodWithKids
  needs_good_with_other_pets      = needsGoodWithOtherPets
  willing_medically_complex = willingMedicallyComplex
  city/state/zip            = city/state/zip
  notes                     = notes

Application Snapshot

On POST /api/adopter/applications:

- If questionnaireAnswers are included in the request:
  - Save/update adopter_questionnaires first
  - Sync adopter_profiles
  - Save questionnaireAnswers into questionnaire_snapshot_json

- If questionnaireAnswers are not included:
  - Use the adopter's most recent adopter_questionnaires record
  - Copy those values into questionnaire_snapshot_json

- Applications must always store a full snapshot of questionnaire answers so staff can review the information exactly as it existed at submission time.

- Google OAuth2 (optional MVP add-on):

- Authorization:

    - PUBLIC routes for animal browsing
    - ADOPTER routes for applying and viewing own applications
    - STAFF routes for management actions
- CORS:
  - Allow GET, PUT, POST, PATCH, DELETE from Angular frontend

  Updates animal information including:
    status
    foster placement

MVP Workflow Rules (Business Logic)
- Applying:
  - Only authenticated adopters can submit applications
  - Prevent duplicate active applications: same adopter + same animal where status in {SUBMITTED, IN_REVIEW, APPROVED}
- Approval/Denial:
  - Only staff can approve/deny
  - On approve: application.status=APPROVED; optionally set animal.status=ADOPTION_PENDING
  - On deny: application.status=DENIED (animal status unchanged unless it was ADOPTION_PENDING, then revert)
- Adoption finalize:
  - Only staff can finalize
  - Once adopted, animal cannot be adopted again; status locked to ADOPTED
- Moves:
  - Only staff can move placement
  - Shelter move sets current_shelter_id and clears foster_user_id
  - Foster move sets current_foster_user_id and may clear shelter_id (or keep shelter as owning org; choose one and enforce consistently)
- Event log:
  - Every intake/move/status change/application submit/approval/denial/adoption writes an animal_event row

Testing
Goal:
Verify API request/response behavior, validation, and authorization rules without starting a real HTTP server.

Purpose
Add a backend testing suite that runs locally and verifies:
1) business logic (service-layer rules and state transitions)
2) API behavior (request/response, validation, authorization) using mocked HTTP calls
3) persistence and constraints against a real MySQL 8 instance to catch database-specific behavior early

Testing Stack
- JUnit 5 (unit and integration testing)
- AssertJ (assertions)
- Mockito (mock dependencies in unit tests)
- Spring Boot Test (test runner and configuration support)
- MockMvc (mock HTTP requests to controllers)
- spring-security-test (authorization and role testing)
- Testcontainers (MySQL 8 container for integration tests)

Test Categories

1) Unit Tests — Service Layer

Goal:
Verify business logic and workflow rules without starting the full Spring application or web server.

Target Services:
- ApplicationService
- AdoptionService
- AnimalService
- QuestionnaireService

Key Business Rules to Test:

ApplicationService
- Prevent duplicate active applications
  A user cannot create another application for the same animal when an existing application status is:
  {SUBMITTED, IN_REVIEW, APPROVED}
  A user cannot create an application for an animal that has been adopted

- Questionnaire snapshot behavior
  Every application must store a full questionnaire snapshot.

  If questionnaire answers are included in the request:
  - Save or update adopter_questionnaires first
  - Sync adopter_profiles
  - Save questionnaire snapshot in questionnaire_snapshot_json

  If questionnaire answers are not included:
  - Load the latest adopter_questionnaires record
  - Copy values into questionnaire_snapshot_json

AdoptionService
- Adoption cannot be finalized without an existing application.
- Adoption cannot be finalized if animal.status == ADOPTED.

On successful adoption finalization:
- Create an adoptions record
- Update animal.status to ADOPTED
- Create an animal_event with type ADOPTED

AnimalService (movement and status logic)

Shelter Move
- Update current_shelter_id
- Clear current_foster_user_id
- Create animal_event with type SHELTER_MOVE

Foster Move
- Update current_foster_user_id
- Maintain consistent shelter ownership rule
- Create animal_event with type FOSTER_MOVE

Status Changes
- Prevent invalid transitions (example: ADOPTED -> IN_FOSTER)
- Create animal_event with type STATUS_CHANGE

QuestionnaireService
On questionnaire submission:
- Upsert adopter_questionnaires record
- Update users.phone if provided
- Synchronize overlapping fields into adopter_profiles

Definition of Done (Backend Testing)

The backend testing suite is considered complete when:

- Service-layer unit tests cover all critical business rules and workflows.
- Controller tests verify authorization rules, request validation, and successful endpoint flows.
- At least one full integration test validates the adoption workflow using a real MySQL database through Testcontainers.
- All tests run locally using a single command (`./mvnw test`) without external dependencies such as AWS.

Acceptance Criteria
- Public can browse animals and filter results
- Adopter can register/login, fill profile, and submit an adoption application
- Staff can:
  - Create/update animals (intake)
  - Move animals to shelter or foster
  - View applications and approve/deny
  - Finalize adoption and mark animal adopted
- Database stores:
  - Users with hashed passwords
  - Roles
  - Animals + photos
  - Applications
  - Event history for each animal
- API follows REST conventions, returns consistent error messages for invalid actions (e.g., adopting an already-adopted animal)
- Basic documentation:
  - README setup instructions
  - API endpoint list
  - Entity overview
  - How to run frontend/backend locally

Implementation Notes
- Prefer DTOs for request/response; do not expose entities directly
- Use service layer for business rules; controllers should stay thin
- Validate input with javax/jakarta validation annotations
- Use soft deletes for animals if deletion is required (INACTIVE)
- Keep “AVAILABLE” as a derived concept for browsing rather than a stored status if desired
- Photos are stored in AWS
- Use environment variables with default values intended for application running locally
- Follow best security practices pertaining to PII and user information
- Use a global exception handler and implement graceful error handling on both frontend and backend
